1 ЗАВДАННЯ
В цій лабораторній роботі необхідно внести зміни в документ Vision & Scope, який ви створили минулого семестру, а саме в розділ 3 Scope and Limitations. В цьому розділі необхідно описати: 
3.1 Scope of Initial Release — опис скоупу всього проекту, разом із серверною частиною, мобільним і Web клієнтами та IoT. Описати роботу всієї системи. 
3.2 Scope of Subsequent Releases — продумати як в майбутніх релізах системи можна інтегрувати штучний інтелект. Треба описати 3 ключові аспекти роботи з ШІ: 
- застосування — як ШІ може покращити роботу вашої системи; 
- збирання даних — які дані і яким чином можна збирати для подальшого навчання моделей; 
- навчання/використання — чи треба навчати модель самостійно, донавчати існуючу, чи використовувати готовий API сервісу ШІ. Продумайте як краще за все використати дані які ви можете збирати. 
3.3 Limitations and Exclusions — з описаного функціоналу (з обох попередніх пунктів), які є неочевидні обмеження. Який функціонал міг би бути присутній в системі, але навмисно виключений і чому.

2 ХІД РОБОТИ
Назва системи: Програмна система для прогнозування погоди із використанням технологій ІоТ, веб та мобільних клієнтів.
Короткий опис: Система призначена для збору й відображення локальних погодних даних у зручній формі. Вона вирішує проблему відсутності точних прогнозів для конкретної місцевості за допомогою інтеграції простих ІоТ-пристроїв. Ці пристрої фіксують температуру, вологість і тиск, та надсилають дані на сервер для обробки. Користувачі отримують доступ до поточної статистики, графіків та базового прогнозу через зручний веб-інтерфейс або мобільний застосунок.
3 Scope and Limitations
3.1 Scope of Initial Release
У першому релізі система охоплює повний цикл збору, обробки та відображення базових метеорологічних даних із локальних сенсорів користувача. Робота всієї системи побудована на взаємодії чотирьох основних компонентів:
* ІоТ-клієнт: Апаратна частина, побудована на базі плати Arduino, що програмується мовою C++. Клієнт збирає показники температури, вологості та тиску. Зібрані дані періодично надсилаються на сервер за допомогою HTTP POST запитів.
* Серверна частина: Реалізована на Node.js із використанням фреймворку Express. Сервер відповідає за прийом даних від пристроїв та їх збереження у реляційній базі даних PostgreSQL. На цьому етапі сервер виконує лише простий аналіз даних, такий як розрахунок середніх значень, визначення мінімумів/максимумів та побудова базових трендів для прогнозу. Взаємодія з клієнтами відбувається через REST API у форматі JSON.
* Веб-клієнт: Розробляється з використанням HTML, CSS, JavaScript та фреймворку React (або Fetch API). Інтерфейс дозволяє користувачеві переглядати поточні показники погоди, графіки змін за добу, короткий прогноз та реєструвати власні ІоТ-сенсори.
* Мобільний клієнт: Android-застосунок, створений за допомогою Kotlin. Він надає аналогічний веб-клієнту функціонал, але у спрощеному вигляді з адаптивним дизайном для зручного використання на смартфонах.
3.2 Scope of Subsequent Releases
У майбутніх релізах передбачається інтеграція методів штучного інтелекту (ШІ) для переходу від базового розрахунку трендів до точного локального прогнозування мікроклімату.
* Застосування: ШІ значно покращить точність короткострокових прогнозів погоди для конкретної локації. Замість простого розрахунку середніх значень, система зможе аналізувати патерни зміни температури, вологості та тиску, передбачаючи локальні заморозки, тумани або різкі перепади тиску. Крім того, ШІ може виявляти аномалії в роботі ІоТ-сенсорів (наприклад, поломку датчика, якщо показники різко стають нереалістичними).
* Збирання даних: Для навчання моделей використовуватимуться історичні дані, які сервер безперервно збирає та зберігає в базі даних. Основним набором даних стануть часові ряди (time-series), що містять позначку часу, температуру, вологість та атмосферний тиск. Дані будуть накопичуватися місяцями для формування репрезентативної вибірки, що враховує добову та сезонну циклічність.
* Навчання/використання: Оскільки мета системи – локальний прогноз, використання готових глобальних API (на кшталт ChatGPT чи стандартних погодних API) не дасть бажаного результату. Найефективнішим підходом буде донавчання (fine-tuning) існуючої легкої моделі для аналізу часових рядів (наприклад, на базі архітектури LSTM або готових моделей з Hugging Face) безпосередньо на зібраних історичних даних конкретного користувача. Навчену модель можна розгорнути у вигляді мікросервісу на Python поруч із основним Node.js сервером, до якого основний бекенд буде звертатися для отримання предикативної аналітики.
3.3 Limitations and Exclusions
Під час проектування системи було навмисно прийнято ряд обмежень для збереження її простоти, фокусу на локальності та низької вартості розробки:
* Відсутність глобальних метеорологічних даних: Система покладається виключно на дані, отримані з власних ІоТ-сенсорів користувача (температура, вологість, тиск). Інтеграція супутникових знімків, радарів опадів або даних про напрямок вітру виключена, оскільки це потребує інтеграції з дорогими зовнішніми сервісами, що суперечить ідеї створення дешевої автономної системи.
* Обмеження початкового прогнозування: У першому релізі прогнозування базується лише на математичних трендах (середнє значення, мін/макс). Складне моделювання атмосферних фронтів виключено через високу вимогливість до обчислювальних ресурсів серверної частини.
* Обмеження апаратної частини: Система не підтримує складні погодні станції. ІоТ-клієнт обмежений базовими сенсорами для платформи Arduino/ESP8266/Raspberry Pi. Це навмисне виключення складного обладнання (наприклад, анемометрів) дозволяє зробити пристрій максимально доступним і простим у самостійному складанні.
