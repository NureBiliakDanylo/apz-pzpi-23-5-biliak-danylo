3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про систему
* Назва системи: ChatGPT (Chat Generative Pre-trained Transformer).
* Розробник: Компанія OpenAI (США).
* Призначення системи:
   * Область застосування: Сфера штучного інтелекту, зокрема обробка природної мови (NLP). Використовується в освіті, розробці ПЗ, аналітиці, маркетингу та копірайтингу, тощо.
   * Користувачі: Спектр охоплює як приватних осіб (персональні асистенти), так і корпоративний сектор (через API та Enterprise-рішення для автоматизації бізнес-процесів).
   * Основні функції: Розуміння контексту запиту, генерація людиноподібних відповідей, написання та дебаг програмного коду, створення складних аналітичних звітів, багатомовний переклад та мультимодальна взаємодія (обробка тексту, голосу та зображень).
3.2 Архітектурний стиль системи
ChatGPT базується на мікросервісній архітектурі з використанням хмарної інфраструктури (Cloud-native).
* Обґрунтування вибору: 
1. Розділення обчислень: Нейромережеві обчислення (Inference) потребують специфічного заліза (GPU/TPU), тоді як веб-інтерфейс та бізнес-логіка працюють на стандартних CPU-вузлах.
2. Гнучкість: Мікросервіси дозволяють OpenAI оновлювати окремі компоненти (наприклад, систему модерації), не зупиняючи роботу всієї платформи.
3. Масштабованість: Можливість горизонтально нарощувати потужність окремих вузлів у пікові години навантаження.
3.3 Основні компоненти системи
* Інтерфейс користувача (Frontend): Платформи на базі React та Next.js, що забезпечують безперебійну взаємодію через мобільні та веб-браузери.
* Серверна частина (Backend/API Gateway): Написана переважно на Python та Go. Виконує роль диспетчера, що керує авторизацією через OAuth 2.0 та протоколами безпеки.
* Рівень бізнес-логіки (Inference Services): Найважливіший рівень, де розгорнуто самі моделі. Тут відбувається токенізація вводу та формування вихідного потоку даних.
* Рівень зберігання даних (Storage Layer):
   * PostgreSQL: Служить «скелетом» системи. Тут зберігаються дані користувачів, налаштування та історія повідомлень.
   * Citus (розширення для Postgres): Забезпечує шардинг (розподіл) даних по багатьох вузлах.
   * Vector DB (Pinecone/Milvus/Qdrant): Для пошуку за змістом та роботи з пам’яттю контексту.
* Допоміжні сервіси: Моделі безпеки (Moderation API), сервіси моніторингу та білінгові системи (Stripe).
3.4 Взаємодія компонентів
Взаємодія в ChatGPT побудована на принципі асинхронного стримінгу та подієво-орієнтованої архітектури. Це дозволяє системі обробляти запити, що потребують значного часу на генерацію, не блокуючи ресурси сервера.
* Опис потоків даних:
   * Request Flow: Запит від клієнта (HTTPS/WSS) надходить на Load Balancer, який перенаправляє його на найменш завантажений вузол API-шлюзу.
   * Context Assembly: Серверна частина звертається до бази даних (PostgreSQL) для отримання останніх 10-20 повідомлень діалогу, щоб модель мала "пам'ять" контексту.
   * Inference Queue: Оскільки ресурси GPU обмежені, запит стає в чергу (наприклад, Redis Streams або Kafka). Оркестратор моделей вибирає запит і передає його на обчислювальний кластер.
   * Token Streaming: Модель генерує текст по одному токену за раз. Кожен токен негайно відправляється назад клієнту через протокол SSE (Server-Sent Events).
* Протоколи взаємодії:
   * gRPC: Використовується для внутрішньої комунікації між мікросервісами (наприклад, між API-шлюзом та сервісом модерації) завдяки високій швидкості та низьким затримкам.
   * WebSockets/SSE: Для підтримки тривалого з’єднання під час генерації тексту.
   * TLS 1.3: Для шифрування всіх даних у дорозі.
* Механізми синхронізації:
Використовується Optimistic Concurrency Control (оптимістичне керування паралелизмом). Оскільки користувач рідко пише два повідомлення одночасно, система перевіряє версію діалогу перед записом у БД, щоб уникнути конфліктів при паралельному доступі з різних пристроїв.
3.5 Масштабованість та надійність
Для ChatGPT масштабованість – це не просто додавання серверів, а вирішення проблеми "вузького місця" в базі даних та GPU-ресурсах. На основі досвіду OpenAI зі статтею Scaling PostgreSQL, архітектура реалізована так:
   * Механізми балансування навантаження:
Використовується дворівневе балансування. L4 Load Balancer (на транспортному рівні) розподіляє TCP-трафік, а L7 Load Balancer (Application layer) аналізує HTTP-заголовки для "розумної" маршрутизації (наприклад, спрямування запитів платних користувачів на пріоритетні кластери).
   * Масштабування бази даних (Шардинг):
Згідно з техблогами OpenAI, вони використовують механізм накшталт Citus для горизонтального масштабування PostgreSQL [3].
      * Принцип: Дані мільйонів користувачів розподіляються між десятками фізичних серверів (шардів).
      * Критерій розподілу: user_id. Це гарантує, що всі чати одного користувача знаходяться на одному вузлі, що дозволяє виконувати SQL-запити без об’єднання даних з різних серверів (cross-node joins), що критично для швидкості.
      * Відмовостійкість (High Availability):
      * Hot Standby: Кожен основний вузол бази даних має "гарячу" репліку в іншому дата-центрі. У разі аварії перемикання (failover) відбувається автоматично за лічені секунди.
      * Circuit Breaker Pattern: Якщо сервіс генерації (GPU) перевантажений, API-шлюз тимчасово "розриває ланцюг", видаючи помилку 429 (Too Many Requests), щоб запобігти каскадному падінню всієї системи.
      * Кешування:
Використовується стратегія Write-through caching у Redis. Коли нове повідомлення записується в Postgres, воно одночасно оновлюється в кеші. Це дозволяє миттєво відображати історію чату при оновленні сторінки, не навантажуючи основну БД.
      * Контейнеризація та хмарна інфраструктура:
Система працює в Microsoft Azure Kubernetes Service (AKS).
         * Auto-scaling: Кількість Pod-ів (контейнерів з додатком) автоматично збільшується при зростанні CPU/GPU навантаження [2].
         * Infrastructure as Code (Terraform): Дозволяє OpenAI розгорнути повну копію всієї інфраструктури в іншому регіоні світу за лічені хвилини у разі глобального збою.
3.6 Приклади програмної реалізації
Оскільки сам бекенд ChatGPT є закритим комерційним продуктом, найкращою демонстрацією його архітектурних принципів є офіційні SDK (набори інструментів для розробників) та відкриті бібліотеки від OpenAI. Вони чітко показують, як реалізована взаємодія між клієнтом та сервером.
1. Реалізація Server-Sent Events (SSE) через офіційний Python SDK [1] 
Цей фрагмент коду з офіційної документації OpenAI демонструє принцип асинхронного стримінгу (про який йшлося в п. 3.4). Замість очікування повної генерації відповіді, клієнт отримує дані частинами (чанками) по мірі їх появи на GPU-кластері:
1. from openai import OpenAI
2. 
3. # Ініціалізація клієнта (взаємодія з API Gateway)
4. client = OpenAI()
5. 
6. # Запит на генерацію з увімкненим потоковим режимом (stream=True)
7. stream = client.chat.completions.create(
8.     model="gpt-4o",
9.     messages=[{"role": "user", "content": "Поясни архітектуру"}],
10.    stream=True, 
11. )
12. 
13. # Обробка потоку даних у реальному часі (імітація "друкування")
14. for chunk in stream:
15.     if chunk.choices[0].delta.content is not None:
16.         print(chunk.choices[0].delta.content, end="")


2. Токенізація на рівні бізнес-логіки (офіційна бібліотека tiktoken [5]) Перед тим як текстовий запит потрапляє до нейромережі, він проходить етап перетворення тексту в масив чисел (токенів). OpenAI відкрила вихідний код свого токенізатора tiktoken. Це ілюструє роботу рівня попередньої обробки даних:
1. import tiktoken
2. 
3. # Завантаження офіційного кодування для моделей GPT-4
4. enc = tiktoken.encoding_for_model("gpt-4")
5. 
6. # Перетворення тексту в токени (те, що бачить модель на рівні Inference)
7. tokens = enc.encode("Привіт, як працює ChatGPT?")
8. print(tokens) 
9. # Вивід: [84125, 4392, 11, ... ]
10. 
11. # Зворотне перетворення (те, що система повертає клієнту)
12. text = enc.decode(tokens)
13. print(text)
