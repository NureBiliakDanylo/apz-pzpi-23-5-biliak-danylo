3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про шаблони (патерни) проєктування
Шаблони проєктування – це високорівневі концепції розв'язання типових проблем, які виникають під час розробки архітектури програмного забезпечення. Їх можна розглядати як перевірені часом креслення, які розробник адаптує під свій проєкт. Основне призначення патернів полягає не в наданні готового коду, а у створенні спільного словника для команди (щоб замість довгого пояснення архітектури сказати "тут ми використаємо Адаптер") та уникненні "винайдення велосипеда" при проєктуванні складних систем.
У класичній книзі "Банди чотирьох" (Gang of Four) виділено 23 основні патерни, які поділяються на три великі групи:
1. Породжувальні (Creational): Вирішують проблеми, пов'язані з безпечним створенням об'єктів. Вони роблять систему незалежною від способу створення, композиції та представлення її об'єктів.
2. Структурні (Structural): Відповідають за те, як класи та об'єкти утворюють більші та складніші структури. Вони допомагають гарантувати, що зміна однієї частини системи не призведе до руйнування іншої.
3. Поведінкові (Behavioral): Визначають алгоритми та способи реалізації ефективної взаємодії, комунікації та розподілу обов'язків між об'єктами.
Патерни застосовуються на етапі проєктування архітектури (System Design), під час рефакторингу старого коду, а також при розробці бібліотек чи фреймворків, де потрібна максимальна гнучкість та можливість розширення функціоналу в майбутньому.
3.2 Опис обраного патерна
Назва патерна: Adapter (Адаптер), також відомий як Wrapper (Обгортка).
Тип шаблона (патерна): Структурний.
Основне призначення: Адаптер вирішує проблему несумісності. Він виступає проміжним шаром, який дозволяє об'єктам із несумісними інтерфейсами успішно співпрацювати. Шаблон конвертує виклики, які посилає один об'єкт, у формат, зрозумілий для іншого об'єкта, приховуючи всю логіку перетворення всередині себе.
Передумови застосування:
* Інтеграція legacy-коду: Коли у вас є стара, але надійна підсистема, яку потрібно підключити до нового коду, але їхні інтерфейси кардинально відрізняються.
* Використання сторонніх бібліотек: Коли ви завантажуєте зовнішню бібліотеку (наприклад, для аналітики чи роботи з БД), інтерфейс якої не збігається з архітектурою вашого додатку, і ви не можете змінити її вихідний код.
* Об'єднання ізольованих класів: Коли є кілька існуючих класів, які виконують схожі функції, але не мають спільного батьківського інтерфейсу, і ви хочете працювати з ними поліморфно.
Переваги:
* Дотримання принципу єдиної відповідальності (Single Responsibility Principle): ви відокремлюєте складний код конвертації даних від основної бізнес-логіки програми.
* Дотримання принципу відкритості/закритості (Open/Closed Principle): ви можете створювати нові класи-адаптери для будь-яких нових сервісів, взагалі не змінюючи існуючий клієнтський код.
Недоліки:
* Збільшення загальної складності програми через необхідність створювати додаткові класи та інтерфейси.
* Можливе зниження продуктивності (дуже незначне, але наявне) через додатковий крок виклику методів через проміжний об'єкт.
3.3 Структура шаблона (патерна)
Відповідно до представленої UML-діаграми (Рис. 1), до структури Адаптера на рівні об'єктів входять чотири основні елементи:
1. Client (Клієнт): Клас, який містить поточну бізнес-логіку програми. Він має посилання target на цільовий інтерфейс.
2. Target (Цільовий інтерфейс): Абстрактний клас або інтерфейс, який описує контракт, за яким звик працювати Клієнт. Він визначає метод +Request().
3. Adaptee (Адаптований клас / Сервіс): Клас, який має корисний функціонал (метод +SpecificRequest()), але абсолютно несумісний інтерфейс з тим, що очікує Клієнт.
4. Adapter (Адаптер): Клас, який успадковує від Target і зберігає всередині себе посилання adaptee на об'єкт Adaptee.
Процес взаємодії відбувається непомітно для Клієнта. Коли клієнтському коду потрібно виконати дію, він викликає метод Request() на об'єкті Adapter, сприймаючи його як звичайний об'єкт Target. Отримавши цей виклик, Adapter виконує точну реалізацію цього методу, просто перенаправляючи виклик на метод SpecificRequest() свого внутрішнього об'єкта adaptee.
  

Рисунок 1 – UML-діаграма


На UML-діаграмі чітко видно взаємодію:
* Спрямована асоціація від Client до Target показує, що Клієнт використовує цей інтерфейс.
* Лінія реалізації/успадкування (зі стрілкою) від Adapter до Target показує, що Адаптер імплементує цей контракт і є його підтипом.
* Спрямована асоціація від Adapter до Adaptee (через посилання adaptee) показує композицію: Адаптер містить екземпляр сервісу.
* Ключова деталь: Пунктирна лінія з текстом adaptee.SpecificRequest() вказує на точну реалізацію методу Request() в Adapter. Вона демонструє, що реалізація методу Request() в Адаптері є просто викликом специфічного методу сервісу.
3.4 Реалізація шаблона (патерна)
Опис реалізації буде виконано мовою програмування Python.
Розглянемо практичний приклад. Уявіть, що наша програма опрацьовує фінансові дані і всюди використовує формат XML. Проте ми вирішили підключити потужну сторонню бібліотеку для побудови графіків, яка приймає лише дані у форматі JSON. Ми використаємо Адаптер, щоб "подружити" їх.
Основні етапи:
1. Створення базових інтерфейсів
Спочатку визначаємо інтерфейс для нашої системи, яка очікує отримати аналітику на основі XML.
# Цільовий інтерфейс (Target)
class AnalyticsLibraryInterface:
    def analyze_data(self, xml_data: str) -> str:
        pass
2. Реалізація конкретних класів
Тепер маємо сторонній сервіс (Adaptee), який працює тільки з JSON, та сам Адаптер.
import json
import xml.etree.ElementTree as ET


# Стороння бібліотека (Adaptee) - Несумісний інтерфейс
class AdvancedJSONAnalytics:
    def display_charts(self, json_data: dict) -> str:
        # Логіка побудови графіка на основі JSON
        return f"Графік успішно побудовано за даними: {json_data}"


# Клас Адаптер
class XMLToJSONAdapter(AnalyticsLibraryInterface):
    def __init__(self, analytics_service: AdvancedJSONAnalytics):
        self._analytics_service = analytics_service


    # Реалізуємо метод, який очікує клієнт (XML)
    def analyze_data(self, xml_data: str) -> str:
        # Етап 1: Конвертація з XML у зрозумілий формат (Словник/JSON)
        root = ET.fromstring(xml_data)
        json_ready_data = {child.tag: child.text for child in root}
        
        # Етап 2: Виклик методу сторонньої бібліотеки
        return self._analytics_service.display_charts(json_ready_data)
3. Демонстрація використання
У клієнтському коді ми працюємо через Адаптер, передаючи звичний нам XML.
if __name__ == "__main__":
    # Наші дані у звичному форматі XML
    my_xml_data = "<data><sales>1500</sales><profit>300</profit></data>"


    # Створюємо екземпляр сторонньої бібліотеки
    json_service = AdvancedJSONAnalytics()


    # Створюємо адаптер, передаючи йому сторонню бібліотеку
    adapter = XMLToJSONAdapter(json_service)


    # Клієнт передає XML, але все успішно працює!
    result = adapter.analyze_data(my_xml_data)
    print(result) 
    # Вивід: Графік успішно побудовано за даними: {'sales': '1500', 'profit': '300'}
Як бачимо, завдяки Адаптеру наша система продовжує працювати з XML, а стороння бібліотека отримує потрібний їй JSON.
